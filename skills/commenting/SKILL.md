# コメント運用スキル

## 目的

コードを読めば動きが追える状態では不十分である。
このスキルは、保守者が「なぜこの実装なのか」「どの制約を守っているのか」「どのコーナーケースを意識したのか」を短時間で判断できる状態を作る。

## 適用タイミング

**必ず適用:**
- 新機能の実装
- バグ修正
- リファクタリング
- 既存コードのレビュー
- 複雑な条件分岐、例外処理、バリデーション、セキュリティ判断を含む変更

**追加で強く推奨:**
- 外部API契約に依存するコード
- ドメインルールを実装するコード
- 一見すると冗長に見える防御コード

## 核心原則

1. **日本語で書く**: プロジェクト既定に従い、コード内コメントは原則日本語で記述する
2. **WHATよりWHY**: コードを読めば分かる説明は書かず、設計理由と制約を書く
3. **必要な場所だけに書く**: 自明な1行ごとの説明は禁止する
4. **将来の読者を前提に書く**: 変更時点の前提、失敗条件、境界条件を残す
5. **コメントと実装を同期する**: 実装変更時にコメントも同時更新する

## コメント配置ルール

### 1. 型・クラス・公開API

- クラス、公開メソッド、公開関数、外部契約となる型には、責務と利用上の前提が分かるコメントを付ける
- 言語標準のドキュメント形式がある場合はそれを優先する
  - C#: XML ドキュメントコメント
  - TypeScript: JSDoc
  - Python: Docstring

最低限含める内容:
- その要素の責務
- 呼び出し側が守るべき前提条件
- 戻り値や副作用のうち、読み手が見落としやすいもの

### 2. 条件分岐・ループ・例外処理

- 分岐条件が業務ルール、セキュリティ制約、障害回避、互換性維持のために存在する場合は、直前に理由を書く
- 早期 return、continue、break が読み手にとって非直感的な場合は、採用理由を書く
- catch 節では、握りつぶしていない理由、再スロー判断、フォールバック理由を書く

形式例:

```csharp
// WHY: 外部送信元の再送で同じ requestId が短時間に重複するため、二重登録ではなく冪等成功として扱う。
if (store.Contains(requestId))
{
    return duplicateResult;
}
```

### 3. データ変換・バリデーション

- 変換ロジックの前提が暗黙的な場合は、その前提を書く
- バリデーションが単なる形式チェックではなく業務ルールを表す場合は、意図を書く

### 4. TODO / FIXME

- `TODO` と `FIXME` は理由なく残さない
- 書式は以下に統一する

```text
TODO: AI 2026-03-26 監査ログ永続化に切り替えた後でインメモリ実装を削除する (#123)
FIXME: AI 2026-03-26 タイムゾーン変換が UTC 前提になっているためローカル時刻入力を拒否する (#124)
```

必須要素:
- 種別 (`TODO` / `FIXME`)
- 担当者名または `AI`
- 日付
- 何を変えるか
- 可能なら Issue 番号

## 記述基準

### 書くべきコメント

- なぜこの実装でなければならないか
- なぜ一見単純な実装を避けたか
- どの入力や障害系を防いでいるか
- 外部仕様、運用制約、互換性要件
- 暗黙前提をコードから即座に読めない箇所

### 書いてはいけないコメント

- 変数代入や単純な if を言い換えただけの説明
- 実装と同じ語を並べただけの説明
- 現在のコードと矛盾する古い説明
- ノイズになる装飾的コメント

## 実装時チェックリスト

コード変更後に必ず確認する:

1. 新規または変更した公開APIに責務コメントがあるか
2. 非自明な分岐に WHY コメントがあるか
3. セキュリティ、整合性、業務制約の理由がコード上に残っているか
4. TODO / FIXME の形式が統一されているか
5. コメントが現在の実装と矛盾していないか
6. 自明な説明コメントを追加していないか

## レビュー時チェックリスト

レビューでは、正しさだけでなくコメント品質も確認する:

1. 変更差分に新しい複雑ロジックがあるのに理由コメントが不足していないか
2. 公開APIコメントが最新の振る舞いを反映しているか
3. コメントが「What」偏重で、設計意図や制約が抜けていないか
4. TODO / FIXME が責任者・日付なしで残っていないか
5. 古いコメントがリファクタリング後に放置されていないか

## 鉄則

```
コメント不足のまま複雑なコードを完了扱いにしない
```

複雑なコードに理由コメントがない場合、その実装は説明責任を満たしていない。
