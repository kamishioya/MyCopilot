# リファクタリングスキル

## 概要

外部の振る舞いを変えずに、コード構造と可読性を改善します。
リファクタリングは革命ではなく、段階的な進化です。

## 使用場面

- コードが理解・保守しにくい
- 関数/クラスが大きすぎる
- コードスメルへの対処が必要
- コード構造のせいで機能追加が困難

## 黄金ルール

1. **振る舞いは変えない** - リファクタリングはコードの動作ではなく構造のみを変える
2. **小さなステップ** - 小さな変更を行い、各ステップ後にテスト
3. **バージョン管理** - 安全な状態の前後でコミット
4. **テストは必須** - テストなしのリファクタリングは単なる編集
5. **一度に一つだけ** - リファクタリングと機能変更を混ぜない

## リファクタリングしてはいけない場面

- 動作しており今後変更されないコード
- テストなしの本番クリティカルコード（先にテスト追加）
- 厳しいデッドライン中
- 明確な目的がない場合

## 主なコードスメルと対策

### 1. 長すぎるメソッド/関数
→ 関連する処理を別関数に抽出する

### 2. 重複コード
→ 共通ロジックを抽出して再利用する

### 3. 巨大クラス（God Object）
→ 単一責任の原則に従いクラスを分割する

### 4. 長いパラメータリスト
→ 関連パラメータをオブジェクト/構造体にまとめる

### 5. フラグ引数
→ 別々のメソッドに分割する

### 6. ネストの深さ
→ ガード節（Early Return）で浅くする

### 7. コメントで説明が必要なコード
→ コードを自己説明的にリネームまたは分割する

### 8. データの塊
→ 常にセットで使われるデータをクラス/構造体にまとめる

### 9. 型判定による分岐
→ ポリモーフィズムまたはストラテジーパターンを適用する

### 10. 過度な結合
→ 依存性注入やインターフェースで疎結合化する

## リファクタリングの進め方

1. **現状を理解する**: 変更対象のコードの動作を正確に理解する
2. **テストを確認**: 既存テストがあるか確認、なければ先に追加
3. **小さな変更**: 一度に一つのリファクタリングを適用
4. **テスト実行**: 変更後にテストを実行して振る舞いが変わっていないことを確認
5. **コミット**: 動作確認後にコミット
6. **繰り返す**: 次のリファクタリングへ
